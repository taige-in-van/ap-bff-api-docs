通知前端契约：通知设置页（Email / WeChat 通道总开关、只读 Primary + Extra 邮箱增删〔OTP 验证〕、 WeChat 通知接收方扫码绑定 / 解绑）+ 站内信（顶栏铃铛未读计数、列表 / 详情、已读回执、Dashboard 摘要）。 字段类型以本 spec 为准。 merchant scope 从会话取、不信请求体；全部 op merchant-current-scoped，均挂 MerchantContextGuard（stale-tab 守卫，无豁免）。 全部端点前置 Merchant.status = ACTIVE 门禁（非 ACTIVE → 403；仅 ACTIVE 商户可用日常能力〔通知设置 / 铃铛 / 站内信中心〕，会话上下文含 INCOMPLETE / PENDING_REVIEW / REJECTED 仅供入驻 / 拒绝页）。

notification-settings

通知设置：Email / WeChat 通道总开关 + 接收方管理（Extra 邮箱 OTP 两步、WeChat 扫码绑定会话）。

读通知设置（通道开关 + Email 接收方 + WeChat 接收方）

返回当前商户通知设置聚合：Email / WeChat 通道总开关、只读 Primary（= Owner Primary Email，生命周期归注册登录与认证域、 变更随之即时反映）、已验证 Extra 邮箱列表、WeChat 通知接收方列表（昵称 + 绑定时间，平等、无 Primary/Extra 之分）。 站内信无商户开关（不出现在本聚合）。 生效设置：商户从未配置过通道开关时仍返回 200，两开关取商户已配置值、未配置则取当前部署默认值 （未配置商户实时取当前部署默认、而非某时刻冻结值）；接收方列表独立于开关设置、照常返回已有 Extra / WeChat。

切换 Email / WeChat 通道总开关（关闭仅停投递、配置保留）

添加 Extra 邮箱第 1 步——向候选邮箱发送 OTP

对候选 Extra 邮箱发起 OTP 挑战：服务端规范化（去首尾空格 + 大小写归一）→ 格式校验（非法 → 400，不发码）→ 去重校验（与 Primary 或已有 Extra 重复 → 409 208012）→ 发 6 位验证码到该邮箱（异步外发； 外发受理未确认 → 502/504——非最终发送失败、不新增 Extra、可稍后重试）。 有效期 / 重发冷却 / 尝试上限为部署可配业务参数；冷却窗口内重复请求 → 429。 按多步挑战流程语义不挂幂等键（每次发新挑战，仅使同一候选邮箱〔规范化后〕的既有活动挑战失效 〔superseded，验证同 410〕；不同候选邮箱的挑战互不影响、可并行验证）。 本 OTP 只授予通知投递资格，不创建 / 修改员工登录 ID、不构成登录安全凭据。

添加 Extra 邮箱第 2 步——校验 OTP，通过即生效

校验候选邮箱 OTP：匹配 + 未过期 + 未消费 + 未超尝试上限 → 通过则该邮箱（经验证）加入 Extra 列表并生效。 通过时以当前 Primary 重做去重（挑战有效期内 Owner Primary Email 若已改成该候选邮箱）：与当前 Primary / 已有 Extra 冲突 → 不创建 Extra、返 409 208012。开关配置独立、不受加 Extra 影响（见 updateNotificationChannel）。 OTP 错误分类（按挑战生命周期）：错误但可重试 → 400 204010（剩余尝试数走 context.attemptsRemaining〔整数类型〕、details 不承载）；尝试超限作废 → 423；过期 / 已消费失败 / 被同邮箱新挑战替代（superseded）→ 410 （用户侧提示文案统一「验证失败」、detail 不泄露码值比对信息）。 并发竞态下该邮箱已被添加 → 409 208012。OTP 一次性消费；成功重放例外： 同一挑战已验证成功后以同码重试（首次 201 响应丢失场景）→ 重放首次结果（返已创建接收方，不再走 410）。

发起 WeChat 通知接收方绑定（创建绑定会话、返回二维码）

创建 WeChat 绑定会话并返回二维码：会话创建为 PENDING（expiresAt 与二维码 TTL 同源）后才返回 201； 创建失败 → 500、不返回二维码。 SPA 展示二维码并在会话 TTL 内轮询 getWechatBindSession。用户扫码关注 / 授权后，绑定结果由后端承接、 经轮询可见（同一微信身份跨会话去重合并）。 取消 / 二维码超时 / 扫码失败 → 会话到期即失效，不新增绑定、可重新发起；无物理 CANCELLED 态——用户取消 / 关页仅停前端轮询、不改服务器会话，会话仍 PENDING 且未过 expiresAt 时扫码照常生效。 按多步挑战流程语义不挂幂等键（可安全重发的挑战资源：重复创建各自独立会话、受 TTL 约束自然过期）。 微信侧不可用 → 502/504（无降级伪成功）。会话 TTL 为部署可配业务参数。 与微信快捷登录绑定解耦。

轮询 WeChat 绑定会话状态

轮询绑定会话：pending（待扫码）/ completed（绑定完成，必携接收方）/ expired（可重新发起）。expired 纯时间派生（PENDING 且 now > expiresAt）、非服务端物理状态；用户取消 / 关页只是前端停轮询、不改会话。 轮询不受微信侧可用性影响；expired 由 expiresAt 判定 （纯读不迁移状态）；completed 表示后端已承接绑定完成。会话过期后晚到的扫码不产生绑定 （不新增绑定）；同一微信身份重复扫码不产生新接收方（去重合并）。 会话状态单调不回退。跨商户 / 不存在 → 404。
