商户后台 React SPA 消费的 5 步入驻向导 / 草稿持久化 / Documents 上传 / 提交审核 / 审核状态分流 契约。 本服务 OWNS Application（入驻申请单：Step 1-5 字段 + Documents 引用 + missingFields + 状态机 DRAFT/UNDER_REVIEW/MORE_INFO_REQUIRED/APPROVED/REJECTED/WITHDRAWN）+ initialRefundPin（Step 4 采集、加密持有至激活移交）。 Application 由 AUTH 邮箱激活成功时备为 DRAFT，本模块从 DRAFT 接管 lifecycle、不重建（ONB-FR-01-AC-01）。 字段类型以本 spec 为准，业务规则见 requirements。 幂等/CSRF 口径：① 草稿分步保存（PUT，整体覆盖当前 step、mode=next|draft 两语义）/ Submit（CAS WHERE 上一状态，重放命中目标态返一致）天然幂等故不挂 Idempotency-Key； Documents 上传非幂等（每次产生新 fileId，孤儿由 D-XCUT 回收，绑定在 step 保存时按 docKey 覆盖）。 ② cookieAuth 的非安全方法（POST/PUT）一律挂 X-CSRF-Token（double-submit，fail-closed）。 ③ Add Merchant 创建端点不在本 spec（归 ME POST /merchants，ME-FR-10）；ME addMerchant 原子建 Merchant + link + Application{DRAFT}、前端显式调 ME switchCurrentMerchant 切上下文后，本模块经 GET /onboarding/application 加载已建 DRAFT 续走向导。 ④ 向导渲染所需的后端动态配置（支付方式 / 费率 / Currency×Country 有效组合 / 级联）见 GET /onboarding/config。

onboarding-wizard

5 步入驻向导：加载当前 Application 草稿 + 字段 + missingFields + 状态（GET）；分步保存草稿 + 返回 validationErrors（PUT）；Review 页 Submit 提交审核（POST，DRAFT/MORE_INFO_REQUIRED → UNDER_REVIEW）。(ONB-FR-01/02/03/05)

ONB-FR-01 加载当前 Application 草稿（状态 + 各 step 字段 + missingFields + 绑定文档）

加载当前选中 Merchant 的 Application（AUTH 邮箱激活时已备为 DRAFT，或 ME Add Merchant 时已备新 DRAFT）。 本接口只 LOAD、不创建（ONB-FR-01-AC-01）：Step 1 Contact 预填 AUTH Sign Up 已有字段（已预填字段只读）+ Merchant ID。 merchant 上下文从会话当前选中商户取、不信请求体；多 Application 按当前选中 Merchant 独立返回。 missingFields 仅在 MORE_INFO_REQUIRED 时非空（写权归 30-管理后台审核动作；只承载字段 key + 所属 step，不携带审核原因）。

ONB-FR-01/02/03 分步保存当前 step 字段到草稿（校验失败返 400 + 字段级 validationErrors）

分步保存单个 step（ONB-FR-03 保存点；Back/stepper 不触发后端保存）。两种语义由 mode 区分：next = 校验当前 step 完整后保存 + 推进（校验失败 → 400 + ErrorEnvelope.details〔field-level validationErrors〕，不写持久化 missingFields）；draft = 离开前持久化未完成草稿（放宽必填、仍校验已填字段格式，不推进）。 请求体恰含一个 step key 且须 == path {step}（不一致 → 400）；该 step 整体覆盖已存值，但 write-only 凭据字段（accountNumber / refundPin / confirmRefundPin）省略 = 保留已有值、仅传新值才替换（跨设备恢复后前端无明文、故省略不清空；mode=next 完整性按 accountNumberSet / refundPinSet 判定、不强制重输）。同 step 重复保存覆盖、不挂 Idempotency-Key。 Documents step 保存 = 全量覆盖该 step 文档映射：按 docKey 绑定上传返回的 fileId，省略某 docKey = 移除该绑定（上传成功但未保存 / 移除的 fileId 不绑定，回收归 D-XCUT）。 预填 / 来源锁定 / 继承只读字段（Step 1 已预填项 / Merchant ID / Branch 继承身份字段）服务端忽略写入（保持只读，ONB-FR-01/07）。 仅 DRAFT / MORE_INFO_REQUIRED 可保存；UNDER_REVIEW / APPROVED / REJECTED 草稿冻结 → 409。 initialRefundPin 经 settlement step 的 refundPin + confirmRefundPin 采集，write-only、加密持有、零发布视图（不在任何响应出现，ONB-FR-01-AC-10）。

ONB-FR-05 Review 页提交审核（DRAFT / MORE_INFO_REQUIRED → UNDER_REVIEW）

从 Review 页提交：校验所有字段 + 5 必传 Documents 完整 → 单 D-MERCH 本地事务原子：Application → UNDER_REVIEW（CAS WHERE 上一状态）+ 条件投影 Merchant（M4，防 switcher 漂移）：① 从 DRAFT → Merchant INCOMPLETE → PENDING_REVIEW；② 从 MORE_INFO_REQUIRED → Merchant 保持 PENDING_REVIEW（补料重提；同事务清空 missingFields，见下）→ 一次性提交确认（刷新后转审核状态页）。 application-received 邮件仅首次 DRAFT → UNDER_REVIEW 触发（ONB-FR-06）；MORE_INFO_REQUIRED → UNDER_REVIEW 是补料重提、不重发 application-received，且同事务清空 missingFields（防残留，ONB-FR-02-AC-05）。 事件按源态分：首次发 application.submitted（→ application-received 邮件）、补料重提发 application.resubmitted（不触发邮件）；邮件由 40-05 通知与事件引擎（独立部署）经 outbox 投递（非进程内发信）。 天然幂等：一致重放（已 UNDER_REVIEW 再 Submit）→ 返当前态 200、不重复发事件；非目标源态 / 非法迁移（终态 APPROVED / REJECTED / WITHDRAWN → UNDER_REVIEW）→ 409 208003； 字段或 Documents 不完整 → 400（204001 字段 / 204002 文档）。

ONB-FR-04 上传单个 Document（multipart：file + docKey → FileRef）

上传到 D-XCUT 文件存储返回 FileRef（fileId / fileName / fileSize / uploadedAt）。 前端 + 后端兜底校验：大小（默认 ≤ 2MB 可配 X-07）+ 格式（PNG / JPEG / PDF，按文件实际内容判定、非声明 MIME）白名单 → 不合规返 400。 非幂等：每次上传产生新 fileId（重试不复用、不挂 Idempotency-Key）；本接口只上传不绑定——绑定到 Application 发生在 Documents step 保存（PUT /onboarding/application/steps/documents 按 docKey 写 fileId、同 docKey 覆盖）； 上传成功但未保存 / 重试产生的孤儿 fileId 不绑定、由 D-XCUT 回收。返回的 FileRef 不含文件内容（仅引用 metadata）。
