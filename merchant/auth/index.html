商户后台 注册登录与认证 (20-01 BFF) — 前端契约面 (0.4.26)

Download OpenAPI specification:

商户后台 React SPA 消费的注册 / 登录 / 会话 / 账户安全 / 团队成员 契约。 浏览器持 httpOnly 会话 cookie（登录后 set-cookie），字段类型与校验规则以本 spec 各字段为准。 幂等/CSRF 口径:① 创建类(注册/建成员)与非天然幂等 / 有副作用的写(如改密〔失效会话〕、改成员)挂 Idempotency-Key;真正天然幂等的纯 PUT/DELETE 可不挂;认证类(登录/登出/MFA/扫码)与多步 OTP 挑战类(改邮箱/找回)按流程语义不挂幂等键(每次都应重新认证/校验);例外:找回流程的 factor-verification 步骤挂 Idempotency-Key(该步幂等、重放同键返回同一 resetToken,杜绝网络重试因 OTP 已消费假失败)。 ② cookieAuth 的非安全方法(POST/PUT/DELETE)一律挂 X-CSRF-Token(double-submit,fail-closed);公开端点(security:[])无会话故无 CSRF。

邮箱激活（确认页主动点击触发的 POST；防扫描器 prefetch）

校验一次性验证 token（仅在激活成功时消费;之前失败不消费、可用同 token 重试,避免卡在待验证但 token 已失效）→ 激活账号 → ACTIVE。 token 单次有效、消费后绝不再签发会话（防旧验证链接泄露后被换取会话）。已使用 token 重试 → 410,按 ErrorEnvelope.code 细分:已激活 → AUTH_REGISTRATION_ALREADY_ACTIVATED（FE 跳登录页、用户仍须密码登录）/ token 无效·过期 → AUTH_VERIFICATION_TOKEN_INVALID_OR_EXPIRED（FE 引导重发或重新注册）。两种 410 都不换会话。验证链接的 GET 确认页是前端路由、不激活账号，故不在本 API。 激活成立后按会话建立结果分流：**sessionEstablished=true → 已 set-cookie、前端走应用启动链路（merchantId / 落地页 / 入驻路由经 getCurrentSession + getOnboardingStatus 取，不在本响应返回）；sessionEstablished=false → 激活已成立但本次未建会话、前端跳登录页首次密码登录**。

WeChat 扫码登录 — 轮询（确认成功则签发 BFF 会话）

被动等待「已绑定用户在微信内确认」。未绑定 / 未扫 / 未确认对前端表现一致（WAITING）。 ESTABLISHED 时已 set-cookie（会话有效期按未勾 Remember Me）；账号禁用 / 商户 suspended → REJECTED + reason。扫码登录不追加 MFA。 轮询契约：非法 / 过期 / 未知票据亦返回 200 + status（不用 4xx 表达票据态）——有意为之，避免票据存在性枚举；非法 / 过期票据归一为 EXPIRED，与「未扫」WAITING 反枚举一致。
